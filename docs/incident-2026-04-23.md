# 事故报告：训练作业被 Kubernetes 驱逐（2026-04-23）

## 摘要

2026-04-23，训练作业 `job-cc19-copy-6860`（2 GPU）被 Kubernetes kubelet 强制驱逐（Evicted），状态变为 Failed。根因为该作业所在物理节点的 `vgpaas-kubernetes` 分区可用空间跌破 10% 驱逐阈值，触发 BestEffort Pod 驱逐。

注意：`30201` 是当前 macli 服务部署所在的远端主机端口，用于运维和调查；它不是 `job-cc19-copy-6860` 的作业 SSH 端口。当前 ModelArts API 返回该作业的 SSH 端口为 `30598`。

---

## 环境背景

华为云 ModelArts 训练容器采用以下文件系统布局：

- **overlay（`/`）**：来自物理节点 `vgpaas-dockersys` LVM 分区，每容器有独立 prjquota（2卡=300G）
- **`/cache`**：来自物理节点 `vgpaas-kubernetes` LVM 分区（9.3T），**整块物理分区由该节点所有容器共享，无 per-container 限额**
- **`/temp`**：SFS Turbo NFS（291T），跨所有物理节点共享，持久化

所有 ModelArts 训练 Pod 均以 **BestEffort** QoS 运行（无 resource requests/limits）。

---

## 事件链

### 1. 辅助异常：疑似 OOM / coredump / overlay 写满

作业 `job-cc19-copy-6860` 的事件记录中出现过 coredump 上传记录，ModelArts 失败分析也给出“存储空间不足”。当前 30201 容器内的 overlay（`/`）也接近 300G 配额上限，说明 overlay 写满是该类作业自身失效的合理风险。

但当前可查的 ModelArts 事件无法直接证明 2026-04-23 当日发生了 OOM，或证明 coredump 写满 overlay 是本次 kubelet 驱逐的直接原因。它应被视为辅助异常或并发风险，而不是已确认的驱逐触发条件。

### 2. 关键根因：vgpaas-kubernetes 分区耗尽

K8S 事件日志记录了驱逐事件：

```
kubelet evicted pod job-cc19-copy-6860-worker-0
message: The node was low on resource: ephemeral-storage.
Threshold quantity: 1016021773091, available: 991079804Ki.
```

- 驱逐阈值：`1016021773091 bytes = 946.2 GiB`（= `vgpaas-kubernetes` 总量 `9462 GiB × 10%`）
- 驱逐时可用空间：`991079804 Ki = 945.2 GiB`，比阈值低约 **1.1 GiB**

即 kubelet 监控 `vgpaas-kubernetes` 分区的 `nodefs.available`，当可用量低于总量 10% 时触发驱逐。**这与单个容器的 overlay 使用量无关。**

### 3. 驱逐对象选择

kubelet 在节点磁盘压力下会综合 Pod 是否超过 request、Pod priority、以及相对 request 的资源使用量选择驱逐对象。由于 ModelArts 训练 Pod 的 ephemeral-storage request 为 0，高 ephemeral-storage 用量 Pod 会更容易被选中。

本次事件日志显示 `job-cc19-copy-6860` 主容器使用约 `89845116 Ki` ephemeral-storage，sidecar 使用约 1.5-2.4 GiB，且 request 均为 0。当前没有保留下来足以证明“它是同节点 `/cache` 占用最大作业”的原始 `du` 输出，因此这一点只能作为当时调查结论，不能作为已完全复核的事实。

### 4. OBS 写入满的并发错误

驱逐事件同期伴有 OBS 桶写满的警告/报错，属于独立问题（存储配额用尽），对本次驱逐无直接因果关系。

通过 `macli query` + `du -sh /cache/*/` 对各活跃节点进行了调查：

| 节点（端口）| `/cache` 可用 | 距阈值余量 | 风险 |
|------------|------------|-----------|------|
| A（30204）| 9.3T（健康）| > 8T | 低 |
| B（30205）| ~946 GiB   | ~0 GiB    | **极高**（已在阈值边缘）|
| G（31116）| ~1.2T      | ~254 GiB  | **高**（预测下一个受害者）|
| 其他节点   | > 2T       | 充裕      | 低 |

节点 A 可见的各用户 `/cache` 占用（`du -sh /cache/*/`）分布示例：
- zishan：685G
- zhudi：254G
- MHW05：231G

- 其余：共约 500G

**`/cache` 无 per-container 限额，任意用户写入大量数据都会威胁整个节点上的所有作业。**

## 预测与验证

- **预测**（分析时）：节点 G（端口 31116）距驱逐阈值仅剩约 254 GiB 余量，风险较高
- **验证**：端口 31116 对应的作业 `job-cc19-copy-3c6e` 后续确实因同类 `ephemeral-storage` 磁盘压力被 Evicted，但当前 API 记录的失败时间是 **2026-04-27 18:29 CST**，不是 2026-04-23 当日稍晚。因此该预测方向正确，但原文的验证时间不准确。

## 根因总结

| 层面 | 描述 |
|------|------|
| 直接触发 | `vgpaas-kubernetes` 分区可用空间（945.2 GiB）跌破 kubelet 驱逐阈值（946.2 GiB） |
| 根本原因 | `/cache` 无 per-container 配额，多用户作业写入大量数据（huggingface 模型缓存、训练中间产物）累积耗尽共享分区 |
| 加剧因素 | Pod 的 ephemeral-storage request 为 0，高磁盘用量 Pod 更容易在节点磁盘压力下被选中 |
| 次要事件 | 疑似 OOM / coredump / overlay 写满属于并发风险；当前证据不足以证明其为本次驱逐触发原因 |

## 预防建议

### 短期（运维层面）

1. **定期清理 `/cache`**：在 `scripts/startup.sh` 或 watch 脚本中加入定期清理旧缓存的逻辑（如 `find /cache/hanmo -atime +7 -delete`）
2. **监控 `/cache` 水位**：建议在 `macli server` 的 `/health` 端点加入磁盘用量展示（`vgpaas-kubernetes` 分区）；当节点可用量 < 1.5T 时应主动介入。该功能曾在 `5ac35c9` 中加入，但当前 `main` 已通过 `3ee71b2` 回滚。
3. **优先用 `/temp` 存大文件**：Hugging Face 模型、数据集等大文件放 `/temp/hanmo/`（SFS Turbo，291T，持久化），避免落入 `/cache`

### 中期（架构层面）

4. **启动脚本增加软链**：将 `/cache/huggingface` 改为指向 `/temp/hanmo/cache/huggingface` 的软链接，缓存持久化且不消耗 `/cache` 分区
5. **HF 缓存环境变量**：确保 `HF_HOME=/temp/hanmo/cache/huggingface`，避免每次新作业重新下载
6. **向平台管理员申请**：为 `/cache` 设置 per-container 或 per-user 配额（如果华为云平台支持）

### 长期

7. **跨节点容量均衡**：如果 macli watch 支持在特定节点创建作业，优先选择 `/cache` 余量充裕的节点
